## 0. 一句话

货源是**快速轮转的动态住宅**：IP 几分钟就换、71% 会跨州。当前系统两处设计与这个本质不匹配——

1. **扫描调度**：一整轮里只要有 1 个上游被限流(402)，全局并发就被砍半；而我们每轮都重探约 161 个被限流上游 → 并发**永久被压制**，健康上游被这条“限流尾巴”连累，扫得很慢。
2. **分级**：评分**重了“IP 轮转速度”（既测不准又与问卷无关），轻了“地理一致性”（才是问卷真正要的）** → 真正“一直同州”的好货被压成普通/动态，高级越来越少。

本设计把两件事都改成“**地理为主、问题隔离**”。

---

## 1. 背景与目标

- 用户用途：问卷 / 登录。真正在乎的是**出口地理稳定（同州/同城、不跨州）**，**不在乎同一个 IP 是否长期不变**（货源决定了 IP 必然轮转）。
- 但一个 IP 若**轮转太快（如每 10 分钟）**，问卷做到一半就变，也没用 → 需要一个**最短 IP 持有下限**。
- 目标：① 让健康上游的检测不被问题上游拖慢；② 让“高级”真实反映“地理锁定 + IP 至少稳得住一段时间”，而不是被测不准的速度分误杀。

## 2. 线上证据（2026-06-27）

### 2.1 货源本质
- 池子 1779（可扫 1776）。ip_type：unknown 1711 / hosting 52 / proxy 11 / residential 5（机房滞后修复后，机房已从 158 降到 52，且当前 0 条陈旧）。
- 稳定层：dynamic ~1152 / quasi ~459 / static ~165。
- 7 天主导州占比（住宅池）：**≥99% 同州 = 327 条**；其余大多 <80%（爱跨州）。24h 跨州占 ~71%。
- 结论：真“地理锁定”的池子 ≈ 327（~18%），与用户 12 条实测 2/12≈17% 同量级。

### 2.2 扫描调度问题（根因）
- worker `runScanBatch` 末尾：`hadRateLimit = 本轮有任意一个 402`；`true → 并发砍半`，`false → 并发上调`。
- 我们每轮重探约 161 个被限流上游 → **每轮必然 ≥1 个 402 → 并发永久砍半、永不回升**。
- 最近 1 小时：成功 804 / 失败 209，其中 **190 个 rate_limited（402）、涉及 161 个上游**（约 19% 检测在反复探被限流的、又 402）。
- cooldown 119（其中 108 是 rate_limited）；402 退避仅 5 分钟。
- 全轮覆盖：稳定状态约 1.5 小时一轮（今日多次重启把它一度拖到看起来 8-10h，已修正）。

### 2.3 分级问题
- 现评分（`computeStabilityScore`）：地理分(同城50/同州40/跨州递减) + 速度分(按“最短变动间隔”分桶) = 满分 80，≥60 static / ≥40 quasi / <40 dynamic；可用性门：成功率<70%→dynamic、<90%且static→quasi。
- 速度分用“扫描间隔”做输入，但**采样太稀**（90% 上游 24h 才被扫 ~9-12 次，中位重扫 57min）→ 速度根本测不准。
- 24h 始终同州的 488 条里，**只有 160 进 static，327 被压成 quasi**（纯因速度分低）。
- 即用正确的“按 exitIp 真实变化间隔”重算，结论几乎不变（货源轮转极快，扫描间隔≈IP变化间隔）→ 说明**速度分这条路本身就不该是主轴**。

## 3. 设计 A：扫描调度（问题隔离，不拖累健康检测）

**核心原则：把“问题上游”和“健康上游”彻底分车道；健康车道的并发速度只由健康扫描决定，问题车道只花自己的预算。**

1. **AIMD 信号改为“健康车道 402 比例”**（如健康扫描中 402 占比 > 阈值才退避），不再是“全轮有没有任意 402”。被限流尾巴不再压制健康并发。（**第一刀，最高性价比**）
2. **被限流/超时的用指数退避**：5→10→20→40min 封顶（替代固定 5min），让它们离开每一轮，停止持续制造 402 信号。（**第一刀**）
3. **车道分离 + 优先级预算**：在用(快车道 2min) > 健康全扫(主预算) > 重试限流(小预算、低优先)。问题车道不抢健康车道的份额。
4. **健康车道更短超时**（如 8s，可配），慢/死上游别占着 15s 的并发坑；15s 只留给重试/验证。
5. **总预算视角**：所有上游同一供应商账号 = 一个总速率上限；按上面优先级分配，避免把预算浪费在反复 402 上。

预期：并发能正常往上爬、一轮明显变快、健康上游检测不再被尾巴连累。

---

## 4. 设计 B：地理为主的分级

**核心原则：地理一致性为主轴；IP 轮转“速度分”降级或丢弃；再加一条最短 IP 持有下限做下限保护。**

1. **主轴 = 多天滑动窗口的“主导州占比”**（用窗口内全部样本，对稀采样和偶发抖动都鲁棒），替代“24h 内出现几个州”（一次抖动就拉黑、噪声大）。
   - 滑动窗口 = 任何时刻往回看最近 N 天（不是 N 天结算清零）。
2. **住宅档新定义**：
   - 高级纯净 = 住宅 + 主导州 ≥ 阈值（候选 99%）+ **IP 持有下限**满足（至少能稳住 X 分钟）。
   - 普通纯净 = 住宅 + 主导州 90–99%（偏同州、偶尔跨）。
   - 动态住宅 = 住宅 + 主导州 < 90%（爱跨州）。
3. **IP 持有下限的落地**（因全池高频扫不现实）：
   - 在用的：快车道每 2min 扫，本就能看到 <X min 的快变 → 看到就踢出高级。
   - **提取那一刻验证（最强）**：发高级时连上游、隔几分钟看 IP 变没变，变了换下一条（扩展现有漂移检测为“持有验证”）。
   - 一直没人用的：被提取前不影响；一旦提取由“提取验证”兜底。
4. **“连续稳定时长”改义为“连续地理锁定时长”**（持续保持高级级别多久；掉级清零；IP 轮转不清零）。
5. **速度分**：丢弃，或仅保留为“同等地理下的次要排序信号”。
6. 提取/重绑排序：地理优先 + 连续地理锁定时长优先（沿用已部署的 stableSince 排序思路，但语义改为地理）。

## 5. 待用户拍板的参数

| 项 | 候选 | 用户已选 |
|---|---|---|
| 地理窗口 N 天 | 3 / 7 | **3 天** |
| 高级主导州阈值 | 99% / 95% | **99%** |
| 高级是否要求“同城”而不仅“同州” | 是 / 否 | **否（只要同州）** |
| IP 持有下限 X | 30 / 60 分钟 | **30 分钟** |
| 高级是否开“提取时持有验证” | 开 / 不开 | **开** |
| 速度分 | 丢弃 / 留作次序 | **丢弃（如需可留作同等地理下的次要排序）** |

> 参数已于 2026-06-27 全部定稿。高级 = 住宅 + 3天主导州 ≥99% + IP 持有 ≥30min（提取时验证）。

## 6. 分步实现计划（建议各自独立会话，先安全后核心）

- **P1（低风险、零代码或小改、立即见效）**
  - 设置页把“运营商限流(402)退避”调长（如 30min）观察一轮是否变快。（零代码，可秒回退）
  - AIMD 信号改“健康 402 比例” + 限流指数退避。（小改 worker/shared，带单测）
- **P2（分级地理化）**
  - `recomputeStabilityScores` 改为多天主导州占比为主轴；新增窗口/阈值为可调设置。
  - “连续稳定时长”语义改为“连续地理锁定时长”。
  - 先写失败测试复现，再改；code-audit 收口。
- **P3（车道分离 + 提取持有验证）**
  - worker 扫描调度按优先级分车道、独立预算、健康车道短超时。
  - 高级提取时“IP 持有验证”（扩展漂移检测）。

每步：本地 typecheck + 全量测试全绿 → 用户授权 → 远程灰度部署（scp + build + pm2 restart，gateway 不动）→ 线上验证。

## 7. 测试 / 验证

- 单测：AIMD 信号（健康 402 比例触发；纯尾巴 402 不触发）；指数退避序列；地理主导州占比分级（同州快变=高级、跨州=动态、偶发跨州不拉黑）。
- 线上灰度验证：部署后看“一轮耗时”是否下降、并发是否能爬上去、高级数是否升到接近真实地理锁定数（~327）。
- 回滚：每次部署自动备份源码 + 静态目录，可一键还原。

---

## 8. 风险 / 不做

- 风险：提供商按总速率限流，提速空间有限；提速需小步试、盯 402 率。
- 风险：地理为主会让“高级”里 IP 仍会轮转——这是**有意的**（问卷只要地理稳）；用“提取持有验证”兜底“别太快”。
- 不做：追求“同一个 IP 长期不变”（货源做不到）；放宽标准换“名不副实的高级”。
